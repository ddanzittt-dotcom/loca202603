- 검토 범위: `loca202603`(participant app), `loca-dashboard`(manager console), `supabase/migrations`
- 검토 방식: 코드/SQL/화면 흐름 근거 기반. 확인 불가 항목은 `확인되지 않음`으로 표기.

## 0) 감사 대상 요약(확인된 범위)

- 프로젝트명: LOCA
- 서비스 한 줄 설명(코드 기준): 지도 기반 로컬 큐레이션 + 행사 참여(체크인/완주/설문) + 공유/재저장 플랫폼
- 타깃 유저(코드 기준): 일반 기록자/큐레이터 + 행사 참여자 + 운영자(대시보드)

- 앱 실행 방법: `loca202603/README.md` 기준 `npm run dev`, 대시보드 `npm run dev -- --port 5174`
- 테스트 계정: 사용자 제공값 존재(`test@naver.com`, `test1@naver.com`)이나 자동화에 직접 사용된 로그는 본 감사에서 `확인되지 않음`
- KPI 실수치(DAU/WAU/D1/D7): 코드/문서에서 `확인되지 않음` (집계 로직은 존재, 값은 환경 데이터 필요)

## 1. 현재 앱의 실제 핵심 루프 요약

### 1-1. 지금 실제로 유도하는 행동

1. 사용자 진입
   - 홈/공유 URL에서 지도 접근 (`/s/:slug`, `/shared?data=...`)
2. 핵심 행동
   - 행사 지도에서 체크인 → 완주 → 설문 (`SharedMapViewer`, `useEventMapData`)
3. 확장 행동
   - 지도 공유/저장/가져오기 (`share_click`, `map_save`, `map_import`)
4. 운영 측 측정
   - 대시보드에서 퍼널/유입/체류/설문/공유-저장-가져오기 확인

### 1-2. 반복 사용 루프 존재 여부

- 존재: **부분적으로 존재**
  - XP/레벨/배지/수비니어, 체크인 리마인더, 이벤트 종료 알림이 있음
- 한계: **강한 재방문 루프로는 부족**
  - 미션/캠페인 엔진, 파트너 리워드 루프, 외부 푸시 재호출, 실험 인프라가 없음

### 1-3. 플라이휠 구간별 상태

- 1) B2C 반복행동: **부분 구현**
  - 체크인/완주/설문 루프는 있으나 “다음날 다시 돌아올 이유”가 약함
- 2) 데이터 축적: **구현됨**
  - `event_checkins`, `event_completions`, `view_logs`, `map_saves`, `map_lineage` 축적
- 3) B2B/B2G 결제 이유: **핵심 미완성**
  - ROI/예산/정산/캠페인 단위 상품화 도구 부재
- 4) B2B 예산의 B2C 환류: **거의 비어 있음**
  - 파트너 예산이 사용자 리워드/챌린지로 되돌아오는 구조 없음

## 3. "겉보기 게이미피케이션" vs "실제 플라이휠 강화 요소"

### 3-1. 겉보기 게이미피케이션(장식에 가까운 요소)

- 레벨/배지 노출 중심 UI
  - `HomeScreen`의 레벨 카드/배지 배너는 존재하지만, 파트너 캠페인 과금과 직접 연결되는 규칙은 약함
- 공유 이미지 스티커/프레임
  - 확산 보조 수단이지만 “검증 가능한 참여 데이터”와 직접 연결은 낮음

### 3-2. 실제 플라이휠 강화 요소(이미 있는 강한 씨앗)

- 검증 가능한 행동 로그
  - 체크인/완주/설문/체류/공유/저장 이벤트가 축적됨 (`event_checkins`, `event_completions`, `view_logs`)
- 재사용/자산화 구조
  - `map_saves`, `map_lineage`로 공유→저장→가져오기 선순환 추적 가능
- 운영 대시보드 기본 퍼널
  - 참여 퍼널 + 공유-저장-가져오기 퍼널이 이미 구현되어 판매형 리포트의 기반이 있음

### 3-3. 구분 결론

- 현재 강점은 “데이터 수집/집계 기반”이고, 약점은 “상품화 가능한 캠페인 운영체계”다.
- 즉, **레벨/배지를 더 붙이는 것보다** 캠페인/보상/성과정산 구조를 먼저 붙여야 플라이휠이 돈으로 연결된다.

1. 주간 루프 미션

1. 캠페인 템플릿
   - 축제/투어/시민참여별 체크포인트·설문·공지 기본셋
2. 운영자 성과 리포트 패키지
   - 참여수/완주율/재방문/공유확산 + PDF 자동 생성 고도화

### 5-3. 파트너 예산이 다시 유저 경험을 키우는 환류 기능

1. 스폰서 미션 보상
   - 미션 완료 시 쿠폰/굿즈 코드 발급
2. 지역 챌린지 시즌제
   - 기간 한정 미션 + 지역 상점 제휴 보상
3. 보상 회수 데이터
   - 지급→수령→재방문까지 연결 추적

### 5-4. 운영/정산/어뷰징 대응 기능

1. 체크인 서버 검증 강화(필수)
2. 비정상 참여 탐지 대시보드
   - 급격한 체크인 속도/정확도 이상/세션 패턴
3. 예산 한도/정산 로그
   - 캠페인별 예산 상한, 소진 경고, 정산 내역

### 5-5. 데이터/대시보드/실험 기능

1. 퍼널 표준 이벤트 스키마 v2
2. 코호트 리텐션(D1/D7/W4) 추가
3. 실험/세그먼트 필터
   - 리워드 정책, 카피, 반경 정책 실험 가능화

## 6. 핵심 지표 제안

### 6-1. 유저 측 지표

- Activation rate: 방문→첫 체크인 전환율
- W1 repeat rate: 7일 내 2회 이상 참여 비율
- Save-to-revisit rate: 저장 후 7일 내 재방문율

### 6-2. 미션/챌린지 측 지표

- Mission start rate / completion rate
- 단계별 이탈률(체크포인트 1→N)
- 미션 완료 후 다음 미션 진입률

### 6-3. 리워드 측 지표

- Reward claim rate(지급 대비 실제 수령)
- Reward-to-repeat rate(보상 수령 후 재방문)
- Reward cost per retained user

### 6-4. 파트너 측 지표

- Cost per completion / Cost per verified participant
- Campaign ROI proxy(목표행동 달성 단가)
- Organization renewal intent 지표(재집행률, 재계약률)

### 6-5. 플라이휠 선행지표

- `share_save_rate`, `save_import_rate` (이미 일부 구현)
- 체크인 검증 통과율(강화 후)
- 스폰서 미션 참여율→보상 수령률→재참여율 연결 곡선

## 7. 마지막 결론

### Q1. 지금 LOCA의 가장 큰 병목은 정말 게이미피케이션 부족인가?

- **부분적으로 맞지만, 본질 병목은 그보다 앞단의 “상품화 가능한 운영/성과 구조 부족”이다.**
- 현재는 게이미피케이션 요소(XP/배지)는 있으나, B2B가 돈을 내는 근거(ROI/정산/검증 신뢰)가 약하다.

### Q2. 맞다면 어떤 종류의 게이미피케이션이 핵심인가?

- 장식형(포인트/배지 추가)이 아니라 **검증형 캠페인 미션**이 핵심이다.
- 즉, “행동 증빙 → 성과 리포트 → 예산 환류”를 연결하는 미션/보상 설계가 필요하다.

### Q3. 아니라면 더 본질적인 병목은 무엇인가?

- **성과 증명의 제품화 부족**:
  - 서버 검증 강도, ROI/비용 지표, 정산/예산, 운영 자동화가 부족함
  - 이 상태에서는 B2B/B2G 확장이 영업 의존으로 남고 반복 매출이 어렵다.

### Q4. 현재 상태에서 가장 먼저 고쳐야 할 한 가지는 무엇인가?

- **P0: 서버 체크인 검증 강화(거리/정확도 기반) + 검증 실패 로그화**
  - 이유: 참여 데이터 신뢰를 먼저 세워야 이후 미션/리워드/ROI가 모두 의미를 갖는다.

## 부록: 이번 감사에서 확인된 강점(유지 권장)

1. 참여 데이터 축적 구조는 이미 좋다
   - `event_checkins`, `event_completions`, `view_logs`, `map_saves`, `map_lineage`
2. 재사용 퍼널을 볼 수 있다
   - 대시보드 `fetchShareSaveImportFunnel`
3. 협업 운영 기반이 생겼다
   - 이벤트 협업 역할/승인/운영자 노트(RLS + 승인요청)

이 강점을 유지한 채, “검증 신뢰 + 캠페인 상품화 + 예산 환류” 3축을 붙이면 플라이휠의 수익 연결이 가능해진다.
