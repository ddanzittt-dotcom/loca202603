# LOCA · 탐색 탭 리디자인 구현 브리프

## 배경

LOCA 탐색 탭(`SearchScreen.jsx`)의 **시각 디자인 + 일부 정보 구조**를 재구성하는 작업입니다. 단순 시각 정돈(홈/내 지도 탭처럼)이 아니라, **섹션 1개 제거 + 검색 풀스크린 모드 신설**이 포함되어 평소보다 한 단계 큰 변경입니다.

## 작업 스코프

**범위 안**
- 헤더 압축 (다른 탭과 통일)
- EXPLORE 히어로 다크 카드 해체 → 인사 + 큰 검색바 분리
- 인기 검색어 칩 (검색 진입 보조)
- "둘러보기 좋은 지도" (PICKED) 섹션 **제거**
- "함께 남긴 지도" → **"모두의 지도"** 라벨 변경
- "근처에서 열리는 지도" → **"근처에서 기록해볼 만한 행사"** 라벨 변경
- 모두의 지도 카드 시각 강화 (미니맵 + 단일 CTA)
- 근처 행사 카드 위계 정돈 (상태 뱃지 3종)
- **검색 풀스크린 모드 2종 신설** (빈 입력 / 입력 중)

**범위 밖**
- 검색 결과 페이지 (전체 결과) 디자인
- 검색 API 새로 만들기 (기존 endpoint 활용)
- 다른 탭 변경
- 데이터 모델 변경

## 3개 화면 상태 다이어그램

```
A. 기본 상태
   ↓ (검색바 탭 / focus)
B. 검색 활성 · 빈 입력
   ↓ (입력 시작 · 1자 이상)
C. 검색 활성 · 입력 중
   ↓ (× 클리어)
B. 검색 활성 · 빈 입력
   ↓ (취소)
A. 기본 상태
```

**상태 관리**: `searchMode: 'idle' | 'active'` + `query: string`
- A: `searchMode === 'idle'`
- B: `searchMode === 'active' && query.length === 0`
- C: `searchMode === 'active' && query.length > 0`

검색 결과 페이지(전체 결과)는 별도 라우트로 분리. 이 작업의 범위는 **인라인 미리보기까지**.

마크업:
```jsx
<div className="search-active-bar">
  <div className="search-active-input">
    <SearchIcon />
    <input
      placeholder="지도, 에디터 검색"
      autoFocus
      value={query}
      onChange={(e) => setQuery(e.target.value)}
    />
    {query.length > 0 && (
      <button className="clear-btn" onClick={() => setQuery('')}>
        <XIcon />
      </button>
    )}
  </div>
  <button
    className="cancel-btn"
    onClick={() => {
      setSearchMode('idle');
      setQuery('');
    }}
  >
    취소
  </button>
</div>
```

> **중요**: 클리어(×) 버튼과 취소 버튼은 **다른 동작**.
> - `×` → input 안의 텍스트만 삭제 (검색 모드 유지, 즉 B로 복귀)
> - `취소` → 검색 모드 자체 종료 (A로 복귀)

## C 화면 상세 사양 (검색 활성 · 입력 중) ⚠️ 핵심

### 카테고리 2개만 노출 (장소 제외)

> **중요**: 결과는 **지도 + 에디터** 2개 카테고리만. **장소 카테고리를 추가하지 마세요**.

이유: LOCA에서 장소는 *지도 안에 속하는 마커*이지 독립 콘텐츠가 아님. 사용자가 "성수"를 검색하는 의도는:
- 큐레이션된 지도 발견 → **지도** 카테고리
- 관심 에디터 팔로우 → **에디터** 카테고리

개별 장소(예: "대림창고 성수점")가 필요하면 해당 지도에 들어가서 보면 됨. 결과를 카테고리별로 분산시키는 것보다 의미 있는 단위(지도)로 모아주는 게 탐색 경험에 유리.

### 검색어 하이라이팅

```js
function highlight(text, query) {
  if (!query) return text;
  const escaped = query.replace(/[.*+?^${}()|[\]\\]/g, '\\$&');
  const regex = new RegExp(`(${escaped})`, 'gi');
  return text.replace(regex, '<mark>$1</mark>');
}
```

부분 매치도 잡기 (`seongsu_lover`에서 `seongsu` 매치). 대소문자 무시.

## 결정·질문 필요 항목 (임의 판단 금지)

1. **"모두의 지도" 진입 라우트** — 기존에 별도 라우트가 있는지, 새로 만들어야 하는지?
2. **미니맵 데이터 출처** — 실제 네이버 지도 SDK 사용? 또는 정적 SVG?
3. **미니맵 마커 데이터** — 사용자 위치 기반? 인기 마커?
4. **위치 칩 ("천안 동남구")** — 사용자 GPS 기반? 사용자 설정 기반?
5. **이벤트 데이터 출처** — `maps.category = 'event'` 전체 조회? 별도 endpoint?
6. **이벤트 거리 계산** — 사용자 위치와 이벤트 위치 거리 계산 함수가 이미 있는지?
7. **인기 검색어 데이터** — 글로벌 trending API가 있는지? 없으면 정적 fallback?
8. **최근 검색어 저장 위치** — localStorage / Supabase 어느 쪽?
9. **검색 결과 더보기 라우트** — `/search/results?q=...&category=maps` 같은 별도 페이지가 있는지?
10. **검색 debounce 시간** — 입력 → 결과 호출 간격 (권장 200ms)?
11. **에디터 검색** — 이름 + 핸들 둘 다 매칭? 핸들만?
12. **빈 결과 시 추천** — "다른 키워드 추천" 같은 기능 추가할지?

이 12가지는 프로젝트 정책 / 데이터 의존성에 따라 답이 달라집니다.
